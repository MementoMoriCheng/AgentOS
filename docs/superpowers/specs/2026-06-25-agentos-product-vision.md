## 1. 一句话定位

**做一个"安全可控的 Agent 运行平台"——以窄而深的安全内核为壁垒，承载企业的多个业务场景，卖给企业、私有化部署。**

---

## 2. 产品形态：A2 多场景平台

**壁垒窄而深，场景横向扩，所有场景共用同一套护城河。**

- **核心壁垒（不扩，越磨越深）**：脱敏层 + 权限闸门 + 审计 hash 链 + 可控（资源限额）。
- **业务场景（会扩）**：数据分析/报表**首发**，后续加客服、文档处理、运维等；每个场景 = 一组 Tool + Policy + 脱敏规则，共用下层安全内核。
- **加新场景 = 加 Tool/Policy/脱敏规则，内核零改**（开闭原则）。这正是"壁垒可复用"的体现。

### 2.1 关键区分：壁垒 ≠ 场景
- 我们**不靠**"支持多少场景"赢——场景谁都能抄。
- 我们**靠**"安全内核过硬"赢——脱敏绝不外泄、权限绝对管住、审计绝对可查、资源绝对可控。这是别人抄不走的。
- 因此：**红队/对抗测试是命脉，不是可选。**

## 3. 战略选择（沿用 v2.3 §1.2，确认）

| 维度 | 决策 |
|------|------|
| 项目类型 | Agent 内核 + 调度器（把 agent 当"进程"管理的真 OS） |
| 目的 | 认真的产品 / 创业，长期投入 |
| 目标用户 | 卖给企业，私有化部署 |
| **核心壁垒** | **安全与可控**（四候选维度里最高、最不易被抄） |
| 技术栈 | 混合：Go（内核）+ Python（运行时） |
| 首发场景 | 数据分析/报表（安全卖点最直接、市场大、MVP 可行） |
| LLM 路线 | 云端 DeepSeek 起步 + 脱敏层承担安全核心；生产换本地模型，接口必留 |
| 商业模式 | 卖给企业，按场景/规模付费，私有化部署 |

## 4. 演进路径（从现在到终局）

> **同步注记（2026-06-26）**：状态已与代码对齐。标 ✅ 已实现，📍 已设计未实现，⏳ 未开始。

| 阶段 | 交付物 | 验证目标 | 状态 |
|------|--------|----------|------|
| ✅ **0. 安全内核**（当前代码） | 单 agent + 脱敏 + 权限 + 审计 + Pipeline + EventBus + 并发骨架 | 安全机制能否跑通 | ✅ 已完成 |
| ✅ **0.1 实时仪表盘**（切片 A） | Web 网关 + 实时可观测，让安全机制"看得见" | 强化 demo 卖点 | ✅ 已实现（Gateway + React 控制台，含事件流/脱敏标记/健壮收尾/WS 重连补播） |
| ⏳ **0.5 场景做深** | 真实数据集、对抗测试强化、脱敏增强、`code_exec` 工具 + 硬隔离沙箱 | 把首发场景做穿 | ⏳ **当前最高优先级** |
| ⏳ **1. 多 agent + 第二场景** | agent 生命周期/并发/通信 + 非数据分析场景 | 壁垒可复用 | ⏳ |
| ⏳ **2. 企业级** | 多租户、Web 控制台全功能、审批流、私有化打包 | 能卖给企业 | ⏳ |
| ⏳ **3. 生态** | SDK，让企业自定义场景接入安全内核 | 商业化扩展 | ⏳ |

### 4.0 当前位置（2026-06-26）
阶段 0 + 0.1 已落地并通过测试（Go 测试 ~89 个 + Python 测试 ~21 个 + 8 对抗 + 2 架构验证）。**下一步是 0.5**：把数据分析首发场景做穿。0.5 的关键缺口是**硬隔离沙箱 + `code_exec` 工具**——当前 Sandbox 接口仅为 `Type()` 占位、不跑任意代码，而"agent 跑 pandas/统计代码"是数据分析的真实刚需。这是从"安全地读写文件"升级到"安全地跑分析代码"的关键一跳。

### 4.1 警惕的坑
- **别为"多场景"过早抽象**。先把首发场景（数据分析）做穿，再用第二场景验证壁垒复用。
- **V2 架构文档的通用平台元素**（沙箱池、状态外化、Harness 编译层、1万 agent 伸缩）**不在近中期路线**，仅作为"若未来要扩到通用平台需加什么"的远期参考。
- **壁垒必须真过硬**。脱敏/权限/审计/可控的红队测试贯穿全程。
